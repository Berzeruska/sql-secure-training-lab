📖 O que é SQL Injection?

SQL Injection (SQLi) é uma das vulnerabilidades mais antigas e prevalentes na segurança web. Ocorre quando entradas do usuário são inseridas diretamente em queries SQL sem sanitização adequada.

consulta VULNERÁVEL
// ❌ Código vulnerável

$ username = req.body.user
$ password = req.body.pass

$ const query = `SELECT * FROM users
  WHERE username='${username}'
  AND password='${password}'`

⚠ Entrada não sanitizada!
ataque em ação
// Atacante digita:

admin' OR '1'='1' --

// Query resultante:
SELECT * FROM users
WHERE user='admin'
OR '1'='1' --'
AND pass='qualquer'

✗ Login bypassed!
🧪 Esta é uma implementação REAL com banco de dados MySQL. Todas as queries são executadas e registradas em log.

🛡️ Técnicas de Defesa

As contramedidas certas eliminam completamente o risco de SQL Injection. Veja cada técnica com exemplos de código.

✅ Prepared Statements

A principal e mais eficaz defesa. Separa completamente o código SQL dos dados do usuário — tornando injeção impossível.

❌ Código Vulnerável
// Node.js com mysql2 — ERRADO
app.post('/login', (req, res) => {
  const { user, pass } = req.body;

  // ❌ Interpolação direta
  const sql = `SELECT * FROM users
   WHERE username='${user}'
   AND password='${pass}'`;

  db.query(sql, (err, rows) => {
    // ...
  });
});
✅ Código Seguro
// Node.js com mysql2 — CORRETO
app.post('/login', (req, res) => {
  const { user, pass } = req.body;

  // ✅ Prepared statement com ?
  const sql = `SELECT * FROM users
   WHERE username=?
   AND password=?`;

  db.query(sql, [user, pass], (err, r) => {
    // Dados tratados automaticamente
  });
});

✅ Por que funciona: O banco de dados pré-compila a query SQL e trata os parâmetros como dados puros, nunca como código. Mesmo que o usuário insira '; DROP TABLE users--, isso seria tratado como uma string literal — não como SQL executável.

Outros exemplos de linguagens
PHP (PDO)
$stmt = $pdo->prepare(
  "SELECT * FROM users
   WHERE username=:user"
);
$stmt->execute(['user' => $username]);
$result = $stmt->fetchAll();
Python (psycopg2)
# PostgreSQL
cursor.execute(
  "SELECT * FROM users
   WHERE username = %s",
  (username,)
)
result = cursor.fetchall()
Java (PreparedStatement)
String sql = "SELECT * FROM users
 WHERE username=?";
PreparedStatement ps =
  conn.prepareStatement(sql);
ps.setString(1, username);
ResultSet rs = ps.executeQuery();
C# (.NET)
var cmd = new SqlCommand(
  "SELECT * FROM users
   WHERE username=@user", conn);
cmd.Parameters.AddWithValue(
  "@user", username);
var reader = cmd.ExecuteReader();
🏗️ ORM (Object-Relational Mapping)

ORMs abstraem queries SQL e usam prepared statements internamente. A maioria elimina SQLi por design.

Sequelize (Node.js)
const user = await User.findOne({
  where: {
    username: username, // ✅ seguro
    password: password
  }
});

// Sequelize gera internamente:
// SELECT * FROM users
// WHERE username=? AND password=?
Prisma (Node.js/TypeScript)
const user = await prisma.user.findFirst({
  where: {
    username: username,
    password: hashedPass
  }
});

// ✅ 100% seguro contra SQLi
SQLAlchemy (Python)
user = db.session.query(User)
  .filter_by(
    username=username,
    password=password
  ).first()

# ✅ Parametrizado automaticamente
Hibernate (Java)
User user = session
  .createQuery(
    "FROM User WHERE
     username=:u", User.class)
  .setParameter("u", username)
  .getSingleResult();
🔍 Validação e Sanitização de Entrada

Defesa em camadas: validar formato antes de usar. Não substitui prepared statements, mas adiciona proteção extra.

❌ Sem validação
app.get('/user/:id', (req, res) => {
  const id = req.params.id;
  // ❌ Nenhuma validação!
  // id pode ser: 1 OR 1=1
  db.query(
    `SELECT * FROM users WHERE id=${id}`
  );
});
✅ Com validação
app.get('/user/:id', (req, res) => {
  const id = req.params.id;

  // ✅ Validar formato
  if (!/^\d+$/.test(id)) {
    return res.status(400).json(
      { error: 'ID inválido' }
    );
  }
  // ✅ + prepared statement
  db.query('SELECT * FROM users WHERE id=?', [id]);
});
Lista de verificação de validação
Princípio do mínimo privilégio ▾
O usuário do banco de dados da aplicação deve ter apenas as permissões necessárias. Ex: uma aplicação de leitura não precisa de UPDATE/DELETE. Isso limita o dano mesmo que uma injeção seja bem-sucedida.
Nunca exibir erros SQL ao usuário ▾
Mensagens de erro detalhadas do banco revelam estrutura, versão e outros dados ao atacante. Use logs internos e exiba apenas mensagens genéricas para o usuário final.
Whitelist de caracteres permitidos ▾
Para campos com formato conhecido (ID numérico, email, CPF), valide com regex rigoroso antes de usar. Rejeite qualquer entrada que não corresponda ao padrão esperado.
Senhas com hash (bcrypt) ▾
// Nunca armazene senha em texto puro!
const bcrypt = require('bcrypt');
const hash = await bcrypt.hash(password, 12);
// Salva hash no banco, não a senha

// Verificação:
const ok = await bcrypt.compare(input, hash);
🧱 WAF (Web Application Firewall)

Uma camada adicional de proteção na borda. WAFs detectam e bloqueiam payloads SQLi antes que cheguem à aplicação.

🔍 Padrões que WAF bloqueia
SELECT * FROM BLOQUEADO
UNION SELECT BLOQUEADO
' OR '1'='1 BLOQUEADO
DROP TABLE BLOQUEADO
admin PERMITIDO
produto seguro PERMITIDO
⚠️ Limitações do WAF
!
WAF não substitui código seguro
Atacantes experientes conseguem bypassar WAFs com encodings, case variation e outros truques. WAF é uma camada, não a solução.
!
Falsos positivos
WAFs muito rigorosos podem bloquear requisições legítimas que contenham keywords SQL em contextos normais.
!
Blind SQLi dificilmente bloqueado
Ataques time-based e boolean-based são difíceis de distinguir de tráfego legítimo.
